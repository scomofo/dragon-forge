## Context

Hatchery combines economy spend, random outcome selection, pity counters, dragon creation, shiny state, element soft pity, and duplicate XP grants. Shared Save, Economy, Dragon, and Authored Content ADRs cover the foundation, but they do not decide which service owns RNG, pity mutation, duplicate handling, or rollback-safe pull settlement.

## Decision

Hatchery will be implemented as a `HatcheryService` configured with immutable authored pull tables and explicit dependencies. It owns pull orchestration and pity counter mutation, but it does not own currency, dragon record mutation, or save file commit.

```gdscript
class_name HatcheryService
extends RefCounted

func configure(save_service: SaveService, economy_ledger: EconomyLedger, dragon_progression: DragonProgressionService, pull_table: HatcheryPullTable, rng_factory: HatcheryRngFactory) -> void
func preview_pull(snapshot: SaveSnapshot, pull_id: StringName) -> HatcheryPreviewResult
func execute_pull(pull_id: StringName, source_id: StringName) -> HatcheryPullResult
```

`execute_pull()` opens one SaveTransaction and stages:

1. EconomyLedger Scrap spend.
2. RNG outcome from an injected or transaction-scoped `RandomNumberGenerator`.
3. Rarity pity and element soft-pity updates.
4. Dragon creation through DragonProgressionService, or duplicate XP grant through DragonProgressionService.
5. Pending semantic events for hatch result and progression results.

No hatch event, dragon-added event, pity event, shiny event, or duplicate-XP event publishes until Save / Persistence commit succeeds.

## Authored Data

```gdscript
class_name HatcheryPullTable
extends Resource

@export var pull_id: StringName
@export var scrap_cost: int
@export var rarity_weights: Array[RarityWeight]
@export var element_weights: Array[ElementWeight]
@export var pity_rules: HatcheryPityRules
@export var shiny_rate_basis_points: int
```

Pull tables are immutable runtime content. Runtime pity counters and pull history live in SaveData staged copies only.

## RNG Rules

- Production pulls use a transaction-scoped RNG seed generated by HatcheryService.
- Tests may inject a deterministic seed or scripted outcome provider.
- Hatchery code must not use global `randi()`, `randf()`, or scattered random calls.
- Roll order is stable and documented: spend validation -> rarity roll with pity override -> element roll with soft-pity adjustment -> shiny roll -> duplicate check.
- Roll results must be captured in `HatcheryPullResult` for QA evidence without exposing hidden player-facing pity counters in UI.

| GDD | Requirement |

| `design/gdd/hatchery.md` | Egg pulls, rarity probabilities, Rare+ pity, element soft pity, shiny protocol, duplicate-dragon XP grants. |

## Alternatives Considered

### Hatchery mutates SaveData directly

Rejected. Direct mutation would bypass EconomyLedger, DragonProgressionService, pity rollback, and commit-before-emit rules.

### Global random calls in UI code

Rejected. Global RNG calls are not deterministic in tests and make pity/shiny failures hard to reproduce.

### Precompute pull outcomes outside transactions

Rejected for MVP. Precomputing risks outcome/spend mismatch after rollback or retry.

## Consequences

- Hatchery stories can test pull boundaries deterministically.
- Economy, pity, dragon creation, and duplicate XP either commit together or roll back together.
- Pull UI can preview cost and eligibility without mutating state.
- Balance simulation can reuse HatcheryPullTable and scripted RNG without UI.

## Verification Plan

- Unit tests for rarity pity guarantee at boundary pulls.
- Unit tests for element soft-pity onset and guarantee.
- Unit tests for shiny flag probability using deterministic scripted outcomes.
- Integration tests for exact-balance spend, insufficient Scraps, duplicate XP, save failure rollback, and post-commit event ordering.
