## 2. Convergent findings

1. **The daily-streak display is wrong everywhere it appears (SR2, SR3, FS2, FS3).** Two auditors independently proved the Sprint 5 "+1 fix" (previous sprint's D4) was applied to the wrong value: the victory overlay reads the POST-completion streak and adds 1 again, so it overstates by one on every daily win and claims "×2 applied" on a first daily where the engine applied ×1.0. Separately, neither display site applies the yesterday-seed liveness gate, so a lapsed streak is advertised right up until the win that resets it, and the displayed "×N" never equals the real capped multiplier (1.1–1.5). Root cause is shared: the UI displays raw `save.dailyStreak` arithmetic instead of the value the engine actually computed at `BattleScreen.jsx:666`.

2. **Mirror Admin defeat is routed as a victory (SR1, SR4, FS4).** Three findings converge on the same broken loop: handleSingularityBattleEnd checks only `isMirrorAdmin`, never the outcome, so the defeat overlay's TRY AGAIN sends a loser to the unconditional victory Credits epilogue with celebration music — while the defeat hint above the button tells them to visit the Campaign Map, a third destination that is wrong for every non-campaign battle type.

3. **The Godot singularity arc is simultaneously unreachable and ungated (GL1, CP3, CP5).** Three independent reads found the same structural hole from different angles: the intended gate (`save.singularity_defeated`) is a dead array nothing writes; the ported gating module (`singularity_progress.gd`) is called only by tests, reads browser-shaped keys, and returns the wrong post-completion stage; and the ungated Battle Select roster exposes the final boss — which is only phase 1 of 3 but pays out full completion rewards including the Light Dragon — to a fresh save.

4. **Godot progression numbers have drifted while combat data stayed in sync (CP1, CP2, CP4 vs CP6/SR5).** The Sprint 6 type chart and light sync were propagated correctly (cell-for-cell parity, test-covered), but everything around it — level caps (50 vs 100 vs unbounded), XP scaling (clamped ratio vs flat reward), boss rewards (1.3–2× inflated) — diverges from the browser source of truth with no documenting comment. The drift is concentrated precisely where the Godot test suite has no pinning assertions.

## 3. Decisions

**E1 — Gate the Credits route on actual Mirror Admin victory.** In `handleSingularityBattleEnd` (`src/App.jsx:144-150`), route to `SCREENS.CREDITS` only when `wasMirrorAdmin && loadSave().mirrorAdminDefeated` (the flag is set exclusively on the true-victory path, `BattleScreen.jsx:683-684`); otherwise return to `SCREENS.SINGULARITY` with battle-end music. The handler already calls `refreshSave()`, so reading the flag from the refreshed save is sufficient — no signature change to `onBattleEnd` needed. (SR1 — high)

**E2 — Close the guidance null-gap and the dead-end shop chip.** In `src/playerGuidance.js`: (1) add a campaign-continuation branch immediately above the final `return null` (line 95): when `getAvailableCampaignNodes(save).length > 0` and not all 9 nodes are defeated, emit `{ action: 'CONTINUE', target: 'map' }` naming the next available node. (2) Replace the raw `dataScraps >= 100 || hasCores` trigger (line 57) with an actionability check — `canAffordBuy` on the cheapest BUY item or `canForge` on any recipe (both already exported from `shopItems.js:100-125`) — and move the branch below the forge/fusion/singularity branches so a banked core no longer shadows them permanently. (3) Update `playerGuidance.test.js:85-96` (which currently enshrines the null) to expect the CONTINUE chip, and add a case asserting SPEND REWARDS does not fire at 30 scraps + 1 core. (FS1 — high)

**E3 — Wire the Godot singularity gate to flags that actually get written.** Replace the dead-array check at `hatchery_screen.gd:42-43` with `SaveHelper.count_singularity_defeated(_save) >= 1` (reads mission_flags, which `battle_screen.gd:421-423` really writes) or a `recursive_golem_defeated` mission-flag check for first exposure. In `battle_select_screen.gd:47`, skip EnemyData entries whose `reward_flag` begins with `"singularity_"` unless the prerequisite flags are present, so a fresh save can no longer fight the lv-30 final boss and unlock light from minute one. Delete the never-written `singularity_defeated` array defaults (`save_io.gd:33`, `dragon_progression.gd:39`). (GL1 — high)

**E4 — Enforce the level-50 cap in both Godot leveling paths.** `dragon_progression.gd:70,78`: change 100 → 50. `hatchery_engine.gd:69-71`: add `and dragon["level"] < 50` to the while condition and zero `dragon["xp"]` at the cap, mirroring `hatcheryEngine.js:64-68`. Add a headless test asserting no path can produce level > 50. (CP1 — high)

**E5 — Implement final-boss phase chaining in the Godot battle screen.** The 3-phase data already ships in `data/singularity_bosses.json:26-30` and matches browser stats; `battle_screen.gd` should detect `npc_id == "the_singularity"`, load phases from the JSON instead of the single `TacticalBattle.EnemyData` entry, and only fire the `singularity_defeated` reward flow (`battle_screen.gd:421-428`) after phase 3 falls. Track Mirror Admin + the 7-fragment gate as an explicit, written-down Godot backlog item (pending the scope answer in §5 Q1) — the current state, where one-third of the fight grants the full endgame reward including the Light Dragon, is the worst of both options. (CP3 — high)

**E6 — Make every streak display show the value the engine used.** (1) Capture the multiplier computed at `BattleScreen.jsx:666` (and the effective streak from `dailyChallenge.js:63`) into the `SET_VICTORY` payload; render the overlay badge from that payload as "🔥 Streak bonus ×1.3", only when the multiplier exceeds 1.0 — this fixes both the double-count and the false "×2 applied" on every first/broken streak. (2) Export a `getEffectiveStreak(save)` helper from `dailyChallenge.js` applying the `lastDailyCompleted === yesterdaySeed` gate, and use it for the BattleSelect chip (`BattleSelectScreen.jsx:131-132`) so a lapsed streak shows no flame (or "streak lost") instead of 🔥6. Note this supersedes the previous sprint's D4, which added the +1 to a value that is post-completion by render time. (SR2, SR3, FS2, FS3 — medium)

**E7 — Derive Godot player moves per-dragon.** In `battle_screen.gd:120-122`, resolve the move list as: `save.dragon_loadouts.get(dragon_id)` if present and non-empty, else `DragonData.DRAGONS[dragon_id].move_keys` (which already lists radiant_beam/solar_flare for light). Have `apply_singularity_unlocks` (`dragon_progression.gd:116-131`) seed `dragon_techniques`/`dragon_loadouts` for granted dragons. Remove the global `active_techniques` read or keep it strictly as a legacy fallback for the fire starter. Without this, every dragon in the Godot build fights with Magma Breath and light's offensive identity is dead content. (GL2 — medium)

**E8 — Port the browser fusion tables verbatim and fix the preview.** In `fusion_screen.gd`: (1) replace FUSION_TABLE/UNSTABLE_PAIRS with the browser ALCHEMY/OPPOSING_PAIRS values (`fusionEngine.js:1-29`), normalizing every key through `_pair_key` at table-build time so unsorted keys ("stone+shadow", "storm+shadow", "venom+shadow") can never silently miss again; (2) align stability tiers with the browser (same-element → stable, default → "normal" — Godot currently never emits "normal" despite defining its color); (3) pass `def_a`/`def_b` — not their `base_stats` — to `DragonData.calculate_stats` in `_update_preview` (`fusion_screen.gd:154-155`), which currently shows near-zero stats for every pairing. (GL4 — medium)

**E9 — Fix Godot fusion's handling of granted dragons without diverging from browser parity.** Two changes in `fusion_screen.gd`: add the browser's `level >= 10` eligibility gate to `_rebuild_lists` (76-83), and re-run `DragonProgression.apply_singularity_unlocks(_save)` at the end of `_on_fuse_pressed` so consuming light in a fusion matches what the next load would produce instead of silently vanishing for the session. Explicitly do NOT add a `maxi(existing, result)` level floor — the level-averaging overwrite is browser-parity behavior (`persistence.js:288-297`). (GL3 — medium)

**E10 — Sync the four singularity boss reward pairs to the browser values.** `tactical_battle.gd:90-91,100-101,110-111,120-121` → 100/200, 150/300, 200/400, 500/1000 per `singularityBosses.js:11-67`. If the inflation was deliberate compensation for missing replay/XP scaling, document it with a comment instead — but pick one (§5 Q2); right now regular NPCs match exactly while bosses silently diverge, which reads as drift. (CP4 — medium)

**E11 — Repair and wire singularity_progress.gd.** (1) `singularity_progress.gd:8`: return 0 post-completion to match `singularityProgress.js:14`, and update `tests/test_singularity_progress.gd:41-43`, which currently enshrines the wrong value 3. (2) Add protocol_vulture to `TacticalBattle.EnemyData` (stats at `src/gameData.js:260-274`) and restore the browser's unlock gate on it. (3) Actually call `is_singularity_unlocked` from the nav surfaces that expose SINGULARITY (`campaign_map_screen.gd:8-13`, `singularity_screen.gd:8-13`) — today the module is gameplay-dead and the screen is reachable with zero gating. Coordinate with E3 so the two gates agree. (CP5 — medium)

**E12 — Clamp and wire Godot XP scaling.** Add `ratio = clampf(ratio, 0.25, 2.0)` at `battle_engine.gd:41`, then award XP via `calculate_xp_gain(reward_xp, player_level, npc_level)` at `battle_screen.gd:413-417` instead of flat reward_xp. Add a test asserting `calculate_xp_gain(50, 1, 30) == 100` so the clamp is pinned (the existing tests pass with or without it). (CP2 — medium)

**E13 — Branch the defeat-overlay hint on battleConfig.** At `BattleScreen.jsx:1224-1226`: `dailyNpc` → "Retry the daily card from Battle Select" (a daily loss does not consume the day — completion only fires on victory); `isSingularity`/`isMirrorAdmin` → "Return to the Singularity breach to re-engage"; campaign default → keep the map line, but when `getAvailableCampaignNodes(save).length <= 1` say "Level up your dragon and try again, or pick another opponent in Battle Select" — on a first loss the map literally only offers the identical rematch while Battle Select has every non-boss opponent unlocked. (SR4, FS4 — low)

**E14 — Move Godot save sanitization out of the dead migration branch.** The v<3 branch in `save_io.gd:86-98` is unreachable (the DEFAULT_SAVE merge at 81-83 injects version 3 into every file). Hoist the nested-shape repairs into an unconditional sanitize step — hatchery_state/dragon_levels/dragon_xp must be Dictionaries, mission_flags an Array — executed before `apply_singularity_unlocks` (line 101), so a hand-corrupted save degrades instead of crashing on every launch. If version-gated migrations are ever needed again, read `parsed["version"]` before the merge. (GL5 — low)

## 5. Open questions

1. **Godot endgame scope (E5).** Is the Godot build intended to ship the full browser endgame — 3-phase Singularity, 7 fragments, Mirror Admin — or a deliberately reduced arc? E5 implements phase chaining because the data already ships, but Mirror Admin and fragments are absent entirely with no scope note. A one-paragraph decision in the Godot tree would stop this from being re-flagged every audit.

2. **Singularity boss rewards (E10): sync or rebalance?** Godot gatekeeper rewards are 1.3–2× the browser values while the final boss pays 20% fewer scraps. If this was compensation for the missing XP/replay scaling, say so in a comment and keep it; otherwise E10 syncs to the browser. Which?

3. **Streak banner semantics (E6).** Should the victory badge show the actual multiplier ("Streak bonus ×1.3") or the streak count ("🔥 3-day streak")? E6 assumes the multiplier, since the current copy says "applied" — but the flame-count framing may read better; the engine cap of ×1.5 makes large flame numbers misleading either way.

4. **Venom/light balance pass (SR5).** Venom is the weakest attacker (one 2.0 target, three weaknesses) and light's only weakness is the singularity-locked void — defensively near-unpunishable for most of a playthrough. Options: venom→light 2.0 ("corrodes radiance", currently 0.5), or a second light weakness. Pure design call; the chart is mechanically sound without it.

5. **Fusing singularity-locked dragons (E9).** Should light be excludable from the Godot fusion pick lists entirely, or keep browser parity (consumable, then re-granted)? E9 implements parity-with-regrant; exclusion would diverge from the browser but avoid the "consumed my endgame reward" moment in both builds — if exclusion is preferred, the browser should change too.

6. **Defeat hint: static branches or playerGuidance-driven (E13)?** E13 proposes static per-battle-type copy. Wiring the defeat overlay to `getPlayerGuidance` would keep the two surfaces from drifting again, at the cost of coupling BattleScreen to the guidance module.
